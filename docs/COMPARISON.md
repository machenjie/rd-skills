# Comparison

ChangeForge is best described as a professional change engineering skill mesh. This comparison uses stable categories rather than project rankings, star counts, or claims about specific competitors.

| Category | Typical Shape | Difference From ChangeForge |
| --- | --- | --- |
| Prompt or rules collection | A set of reusable instructions. | Usually lacks route selection, profile boundaries, build/install validation, and release evidence. |
| Agent or subagent collection | Multiple specialized agents or personas. | May provide role separation but often misses shared registries, precision reference loading, and validation suites. |
| Command or plugin framework | Runtime commands and integrations. | Can automate actions, but may not encode professional change workflow, evidence gates, or repair/re-review rules. |
| Task management system | Plans, tasks, and handoffs. | Tracks work but does not necessarily provide engineering risk gates or profile-scoped runtime packaging. |
| Professional change engineering skill mesh | Routed skills, selected references, gates, hooks, validation, install/doctor, examples, and scorecards. | This is the ChangeForge target shape. |

## Comparison Dimensions

| Dimension | ChangeForge Position |
| --- | --- |
| Routing | `change-forge-router` selects the minimum sufficient path and risk gates. |
| Staged workflow | Clarification, inspection, TDD signal, implementation, review, repair/re-review, validation, handoff. |
| Risk gates | Security, API/data, reliability, delivery, quality, documentation, and AI review gates. |
| Profile discipline | `recommended`, `full`, and `dev` keep top-level exposure explicit. |
| Reference loading | Foundation capabilities are selected references, not automatic context expansion. |
| Hook guardrail | Hooks are advisory closure reminders, not a router replacement. |
| Validation/eval/benchmark | Local scripts validate registries, routing, professionalism, hooks, builds, install, and codegen smoke. |
| Install/doctor/uninstall | Runtime outputs are built into `dist/` and managed by installers with manifests. |
| Examples/scorecard | Showcase scenarios and generated scorecard explain evidence and residual risk. |

## Boundaries

Do not use this document to make live market claims, star-rank comparisons, or negative claims about other projects. Keep comparisons categorical and verifiable.
