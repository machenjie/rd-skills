# Information Architecture Checklist

- Select the IA mode and scope the included/excluded pages, modules, routes, and user segments.
- Record current source evidence: nav/routes/pages, labels, roles, empty states, search/filter, support/admin paths, research, graph, memory, and validation freshness.
- Accept, reject, or mark not verified any reused repository graph, project memory, taxonomy, nav pattern, support workflow, or prior validation result.
- Identify primary user tasks and decisions.
- Group pages, modules, and content around tasks rather than tables or services.
- Select organization scheme and rejected alternatives.
- Define hierarchy, labels, navigation placement, max depth, and device-specific pattern.
- Identify entry points, cross-links, breadcrumbs, deep links, search, filters, sort, and faceted recovery.
- Define role-specific visibility and restricted destinations: visible, locked, or hidden with reason and direct-entry behavior.
- Split empty, filtered-empty, first-run, archived, unavailable, no-access, and permission-denied structural behavior.
- Check that support, admin, audit, bulk, diagnostic, and operational tasks remain discoverable.
- Identify content that appears in multiple task contexts and the owning source of truth.
- Confirm labels match user domain language and name any unverified vocabulary risk.
- Map every changed task group, label, nav level, visibility rule, empty state, handoff, search/filter decision, and validation method to evidence/test or residual risk.
- For each mapped IA change, name the validator or manual review artifact, pass target or owner, freshness after the final IA edit, and not-run consequence.
- Define handoff boundaries to user-flow, routing-navigation, interaction-state, permission, component, accessibility, copy, analytics, and frontend-testing owners.
