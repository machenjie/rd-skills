# Requirement Structuring Checklist

- State current behavior with observable evidence or a clear evidence gap.
- State desired behavior as outcome, not implementation mechanics.
- Name the actor who initiates or benefits from the change.
- Name the trigger that starts the behavior.
- Define in-scope surfaces, data, APIs, screens, jobs, or workflows.
- Define non-goals and deferred decisions.
- Record constraints, dependencies, compatibility promises, and rollout limits.
- Identify assumptions that remain after clarification.
- Define the acceptance signal and owner.
- Map each requirement to test or review evidence.
