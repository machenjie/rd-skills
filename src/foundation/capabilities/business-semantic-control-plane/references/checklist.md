# Business Semantic Control Checklist

- Name the selected mode and stage fit.
- State why BSP is required or why it is explicitly skipped.
- Keep scope task-bound; reject project-wide business context dumps.
- Classify every claim as `FACT`, `INFERENCE`, `ASSUMPTION`, `OPEN_QUESTION`, or `MEMORY_SIGNAL`.
- For every `FACT`, cite current source, user-provided source material, owner review, or validation evidence.
- Mark repository graph and project memory as selectors until confirmed by current source.
- Build or update object inventory only for affected terms and owners.
- Build or update rule records with `rule_id`, owner, enforcement layer, reason codes, entry points, effective dating, tests or residual risk.
- Build or update workflow records with allowed and forbidden transitions.
- Map business claims to code placement, validation/golden case, review finding, owner review, or residual risk.
- Record selected and skipped BSP references with rationale.
- Handoff with validation status, rollback note, and residual risk.
